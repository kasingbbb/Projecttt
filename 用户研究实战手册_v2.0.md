# 用户研究实战手册 v2.0

**一句话总结：收集真实故事，而非礼貌回答。**

> "If you're not collecting rich stories, you won't identify opportunities." — Teresa Torres

---

## 🎯 核心原则（5条铁律）

### 1. 只和采取过行动的人聊
❌ "你觉得这个功能怎么样？"
✅ "告诉我你上次使用这个功能的情况。"

**Bob Moesta原则**："我只和已经尝试过改变的人聊。关注'switching'（行为转变），忽略'bitching'（抱怨）。"

### 2. 收集故事，不要观点
❌ "你会使用这个功能吗？"
✅ "带我回到上次你遇到这个问题的时候..."

**关键：** 要具体的时间、地点、情境，不要假设和未来时。

### 3. 观察>询问
❌ 相信用户说的
✅ 看用户做的

**Gustaf Alstromer**："最好的理解方式不是问，而是看。让他们演示日常工作流，找到他们已经习以为常的痛点。"

### 4. 永远不要推销
❌ "嗨，我是X公司CEO，我们做Y，让我给你演示..."
✅ "告诉我你上周的工作情况..."

**Jeff Weinstein**："先听。用沉默让他们打开话匣子。"

### 5. 寻找证伪，不是验证
❌ 证明我的想法是对的
✅ 找到我哪里想错了

**Judd Antin**："我们不验证，我们证伪。很多PM想要证明自己是对的——这是做表演，不是真研究。"

## 🚫 避免陷阱（5个常见错误）

### 1. 引导性问题
❌ "你不觉得X很烦人吗？"
✅ "告诉我上次使用X的情况。"

### 2. 假设性问题
❌ "如果我们做了X，你会用吗？"
✅ "上次你遇到这个问题时，你怎么解决的？"

### 3. 样本量太少
❌ 访谈2-3个人就下结论
✅ 至少7-14人，直到不再有新洞察

**Shaun Clowes**："7-14次访谈之间，你会停止学到新东西。少于7，数据不够；多于14，收益递减。"

### 4. 只听会说话的人
❌ 只听反馈最多的用户
✅ 也要关注沉默的大多数、流失的用户、非用户

**Uri Levine**："最关键的洞察来自流失用户，不是成功用户。"

### 5. 把研究外包给别人
❌ 让研究员访谈，PM读报告
✅ PM、工程师、设计师必须亲自参与

**Marty Cagan**："在我访问过30个客户之前，不允许我做产品决策。那些访问改变了我的人生——我以为我了解客户，其实根本不了解。"

## 📋 快速行动指南

### 阶段1：准备（1天）

**1. 明确研究目标**
回答：我们要学习什么？
- □ 验证问题存在性？
- □ 理解用户行为？
- □ 测试解决方案？
- □ 探索新机会？

**2. 确定访谈对象（谁？）**

**优先级排序：**
1. ⭐⭐⭐ **最近切换者**（最近改变了行为）
2. ⭐⭐⭐ **流失用户**（尝试过但放弃了）
3. ⭐⭐ **重度用户**（建立了复杂工作流）
4. ⭐ **非用户**（应该用但没用）

**招募标准（必须满足其一）：**
- □ 最近3个月改变了相关行为
- □ 最近6个月购买了相关产品
- □ 最近停止使用某个解决方案
- □ 在社交媒体上主动讨论过这个话题

**样本量：** 7-10人（定性），然后决定是否需要定量验证

**预期拒绝率：** 90%（如果接受率太高，说明筛选不够严格）

**3. 准备（不是写脚本！）**

❌ 不要：详细的问题清单
✅ 要：心理框架

**使用Four Forces心理框架（Bob Moesta）：**
```
Push（推力）：什么让他们想改变？
Pull（拉力）：新方案吸引他们什么？
Anxiety（焦虑）：什么让他们犹豫？
Habit（习惯）：什么让他们保持现状？
```

随时根据对话调整，跟随有意义的线索。

---

### 阶段2：执行访谈（60-90分钟/人）

**访谈结构：**

**开场（5分钟）：建立信任**
```
"感谢你花时间和我聊。我想了解你的真实经历，
 没有对错答案，你的任何坦诚想法都对我很有帮助。"

第一个问题（暖场）：
"告诉我上次[相关场景]的情况..."
```

**深度挖掘（40-50分钟）：收集故事**

**使用STAR框架引导：**
```
Situation（情境）：
"带我回到那个时刻，当时发生了什么？"

Task（任务）：
"你想完成什么？"

Action（行动）：
"你具体做了什么？能给我演示一下吗？"

Result（结果）：
"结果怎么样？你感觉如何？"
```

**跟进技巧：**
- **沉默**：问完问题后，等待5秒，不要急着说话
- **重复**："你刚说了X，能多说一点吗？"
- **深挖**："有意思，那个时候你在想什么？"
- **具体化**："'经常'是多经常？能给我一个数字吗？"

**演示环节（20分钟）：观察实际操作**
```
"能给我演示一下你上次是怎么做的吗？"
"假装我不在，展示给我看你的日常流程。"
```

**注意观察：**
- 视线移动
- 停顿和犹豫
- 重复操作
- 情绪变化（皱眉、叹气）

**情感探索（10分钟）：挖掘动机**
```
"那个时候你感觉怎么样？"
"如果那个问题消失了，你的生活会有什么不同？"
"什么是最让你沮丧的？"
```

**Nan Yu原则**："我的目标是用户怎么难受，我也怎么难受。挖掘功能请求背后的负面情绪。"

**结尾（5分钟）：支付意愿测试**
```
"如果我今天能解决这个问题，你愿意为此支付多少？"

进阶版（真实性测试）：
"如果我今天能解决这个问题，你愿意现在就付$20预订吗？"
```

**Jeff Weinstein**："'愿意付费'和'真的付费'之间差距巨大。让他们立即行动，测试真实承诺。"

### 阶段3：快速分析（2-3天）

**不要：** 30页报告、SPSS分析、学术论文
**要：** 快速聚类、可传播洞察、立即可行动

**分析流程：**

**步骤1：提取关键时刻（3小时）**
```
听录音/看笔记，标记：
- 😮 惊讶时刻（没想到的发现）
- 😡 强烈情绪时刻（用户愤怒/沮丧）
- 💡 顿悟时刻（用户突然意识到）
- 🔁 重复模式（多人提到同一件事）
```

**步骤2：便利贴聚类（2小时）**
```
工具：白板 + 便利贴
参与者：PM + 工程师 + 设计师

方法：
1. 每人写下5个最深刻的发现
2. 贴到白板上
3. 移动相似的到一起
4. 给每个簇起个名字
5. 投票选出Top 3

产出：3-5个核心主题
```

**步骤3：创建行为聚类（半天）**
```
不按人口统计（年龄、性别），按行为模式分类

示例聚类：
1. "焦虑的连接者"
   - 行为：必须保持连接，但害怕分心
   - 场景：接送孩子的父母

2. "效率极客"
   - 行为：把车当移动办公室
   - 场景：长途通勤的销售

3. "娱乐优先者"
   - 行为：驾驶很无聊，需要娱乐
   - 场景：Uber司机、长途司机
```

**步骤4：提炼可执行洞察（半天）**
```
产出1：One-Pager（1页纸）
- 3个核心发现
- 每个配1条用户原话
- 2-3条产品建议

产出2：Insight Memes（可传播的洞察）
用隐喻让洞察易于记忆和引用

示例：
"Using the car system feels like trying to text
 on a Nokia in 2024"

产出3：情境画像卡片（视觉化）
不是传统画像，而是：
- 场景：什么时候遇到问题
- 任务：想完成什么
- 痛点：什么阻碍了
- 期望：理想状态
```

**Yuhki Yamashita原则**："目标是'meme化'——把洞察提炼到足够朗朗上口，让高管在会议中引用。"

---

### 阶段4：验证与行动（持续）

**快速验证（如需要）：**
```
定量问卷（300-500人）
- 验证定性发现的普遍性
- 测试优先级排序
- 支付意愿定量测试

问卷原则：
- 基于真实故事改编
- 不要问假设性问题
- 用场景化描述
```

**转化为行动：**
```
Week 4：产品建议
- 基于洞察的3-5个产品方向
- 优先级排序（RICE模型）
- 快速原型草图

Week 5：测试
- 与lighthouse users测试原型
- 收集反馈
- 快速迭代
```

**建立持续反馈流：**
```
不要让研究"一次性结束"

每日：社交媒体监听
每周：竞品评论分析
每月：用户支持工单分析
每季度：lighthouse users深度访谈
```

**Shaun Clowes**："聪明的PM持续浸泡在反馈河流中。设置用户访谈、NPS、竞品信息的流，让它们每天冲刷你。"

## 🛠️ 实用工具箱

### 工具1：访谈前Checklist

**研究目标：**
- [ ] 明确要学习什么（1句话）
- [ ] 确定成功标准（学到X就成功）

**招募：**
- [ ] 招募7-10个"行动者"
- [ ] 预期90%拒绝率
- [ ] 准备用户激励（时间补偿）

**物料：**
- [ ] 录音设备/软件（征得同意）
- [ ] 笔记本（手写，不要一直盯电脑）
- [ ] 访谈框架（Four Forces）
- [ ] 演示任务清单

**心态：**
- [ ] 我是来学习的，不是验证的
- [ ] 我要证伪我的假设
- [ ] 没有愚蠢的用户，只有不够好的产品

---

### 工具2：常见问题应对

**情况1：用户只说"挺好的"**
❌ "哦，那就好。"
✅ "有意思，能带我回到上次使用的时候吗？具体发生了什么？"

**情况2：用户开始提功能建议**
❌ "好主意！我们考虑一下。"
✅ "有意思！是什么让你想到这个？告诉我你遇到了什么问题。"

**情况3：用户问"你们会做这个功能吗？"**
❌ "我们正在考虑。"
✅ "我现在还不确定，但你的经历会帮我们决定。告诉我你为什么需要它。"

**情况4：尴尬的沉默**
❌ 赶紧说点什么
✅ 数到5再说话，给用户思考空间

**情况5：用户说"我不知道"**
❌ "好吧，下一个问题。"
✅ "没关系，那让我换个方式问。上次你[相关行为]的时候..."

---

### 工具3：洞察质量检查

**好洞察的标准：**
- [ ] **惊讶**：这个发现让我意外
- [ ] **具体**：有具体的用户故事支撑
- [ ] **可行动**：能转化为产品决策
- [ ] **可传播**：能用一句话说清楚
- [ ] **可验证**：能设计实验测试

**坏洞察的信号：**
- ❌ "用户想要更好的体验"（太宽泛）
- ❌ "用户喜欢简单的界面"（太明显）
- ❌ "年轻用户更喜欢X"（人口统计分类）
- ❌ "90%的用户说他们会使用"（假设性）

## 📊 成功衡量

### 研究质量指标

**定性（主观但重要）：**
- [ ] 每次访谈至少3个"惊讶时刻"
- [ ] 团队能不看文档复述核心洞察
- [ ] 洞察在团队会议中被引用
- [ ] 工程师说"这改变了我的想法"

**定量：**
- 用户主动分享故事（不需要催促）的比例 > 70%
- "行动者"样本比例 > 60%
- 洞察转化为产品功能的比例 > 30%

### 产品影响指标

**短期（3个月）：**
- 基于研究的产品决策数 ≥ 3
- 测试的新概念数 ≥ 2

**中期（6个月）：**
- 研究驱动的功能上线数 ≥ 1
- 用户反馈验证研究的比例 > 70%
